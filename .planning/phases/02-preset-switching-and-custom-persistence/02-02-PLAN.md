  Check all `fieldDescription` values for the 15 numeric settings on the Configuration tab (`cruiserToFrigate`, `cruiserToDestroyer`, `capitalToFrigate`, `capitalToDestroyer`, `capitalToCruiser`, `cruiserCrewMult`, `cruiserSupplyMult`, `cruiserFuelMult`, `capitalCrewMult`, `capitalSupplyMult`, `capitalFuelMult`, `cruiserBuildCostMult`, `capitalBuildCostMult`, `cruiserDmodCount`, `capitalDmodCount`). Confirm none contain old `[Recommended: ...]` or `[Hardcore: ...]` annotation text. The Phase 1 CSV rewrite should have already removed these, but verify. If any remain, remove them.

  **Part B: Add clean description storage and dynamic rewriting (PresetListener.java)**

  1. Add a class-level field to store original clean descriptions:
     ```java
     private static final Map<String, String> cleanDescriptions = new HashMap<>();
     ```

  2. Add a new method `captureCleanDescriptions()`:
     - Iterates `LunaSettingsLoader.INSTANCE.getSettingsData()`
     - For each entry where `MOD_ID.equals(entry.getModID())` and the fieldID is one of the 15 numeric settings (use the existing `BACKUP_SETTINGS_INT` and `BACKUP_SETTINGS_DOUBLE` arrays from Plan 01 to check membership), stores `fieldID -> fieldDescription` in `cleanDescriptions`
     - Only captures if `cleanDescriptions` is empty (first call) to avoid overwriting originals with already-prefixed descriptions
     - Wraps in try-catch, logs on failure

  3. Add a new method `updateSettingDescriptions(String presetName)`:
     - Gets the preset values map for the given preset name using a helper (see step 4)
     - Iterates `LunaSettingsLoader.INSTANCE.getSettingsData()` by index (need `allData.set(i, replacement)`)
     - For each entry where `MOD_ID.equals(entry.getModID())` and the fieldID exists in `cleanDescriptions`:
       a. Get the clean description from `cleanDescriptions.get(fieldId)`
       b. Get the preset value from the preset values map
       c. Format the hint: integers use `value + "%"` (e.g., `30%`), doubles use `String.format("%.1fx", value)` (e.g., `1.5x`). For "Custom" preset, use no hint (clean description only, since Custom has no fixed values).
       d. Prepend: `"[Preset: " + formatted + "] " + cleanDesc`
       e. Construct a NEW `LunaSettingsData` with the modified description. Use exact constructor parameter order (10 params): `modID, fieldID, fieldName, fieldType, fieldDescription, defaultValue, secondaryValue, minValue, maxValue, tab`. Get all values from the existing entry's getters EXCEPT `fieldDescription` which uses the new value.
       f. Replace the entry in the list: `allData.set(i, replacement)`
     - Wraps in try-catch, logs on failure

  4. Add a helper method `getPresetValuesForName(String presetName)` returning `Map<String, Object>`:
     - Switch on presetName: "Vanilla" -> VANILLA_VALUES, "Sirix Recommended" -> RECOMMENDED_VALUES, "Sirix Hardcore" -> HARDCORE_VALUES, "Custom" -> empty map (no hints for Custom)
     - Filter out factionBlacklist from the returned map (only numeric settings should get hints)
     - Return the filtered map

  5. Modify `settingsChanged()`:
     - Call `captureCleanDescriptions()` BEFORE `updateActivePresetIndicator()` (early in the method, captures originals on first run)
     - Call `updateSettingDescriptions(currentPreset)` AFTER `handleLoadPreset()` and the backup call (at the end of the method, after all value changes are applied)

  6. Also call `captureCleanDescriptions()` and `updateSettingDescriptions()` from a new initialization path. The issue is that `settingsChanged()` only fires on save. For initial load, we need descriptions set up when the settings panel first opens. Add a static initializer or call from the constructor:
     - In the `PresetListener` constructor, after `super()` (if any), add:
       ```java
       // Deferred: descriptions are captured on first settingsChanged call
       // since SettingsData may not be populated yet at construction time
       ```
     - This is acceptable: descriptions will be correct after the first save. On initial open, they show the clean CSV descriptions without hints. After the user saves once, hints appear. Per research: "The descriptions update when the user navigates away from and back to the Configuration tab." This is expected behavior.

  **Important pitfall avoidance:**
  - Always use `cleanDescriptions.get(fieldId)` as the base, NEVER `entry.getFieldDescription()` for building the new description. This prevents prefix accumulation (Pitfall 4 from research).
  - The `LunaSettingsData` constructor has 10 positional parameters. Double-check the order against the research: `(modID, fieldID, fieldName, fieldType, fieldDescription, defaultValue, secondaryValue, minValue, maxValue, tab)`.
  - When preset is "Custom", call `updateSettingDescriptions("Custom")` which uses an empty values map, so descriptions revert to clean originals (no `[Preset: ...]` prefix). This is correct -- Custom has no fixed values to hint.
